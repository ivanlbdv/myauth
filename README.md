# MyAuth - система аутентификации и авторизации

## Описание системы управления ограничениями доступа

В системе MyAuth реализована **ролевая модель контроля доступа (RBAC)** с детализированной настройкой разрешений для ресурсов. Ниже описано, как устроена схема управления ограничениями доступа.

---

## Основные сущности системы

1. **User (Пользователь)**  
   - Основная учетная запись в системе.
   - Имеет email, ФИО, пароль (хранится в зашифрованном виде), статусы (`is_active`, `is_staff`, `is_superuser`).
   - Может быть связан с одной или несколькими ролями через модель `UserRole`.

2. **Role (Роль)**  
   - Логическая группа прав (например, «администратор», «пользователь»).
   - Каждая роль имеет уникальное имя и описание.
   - Примеры: `admin`, `user`.

3. **Permission (Разрешение)**  
   - Атомарное право на выполнение действия (например, «просмотр постов», «удаление пользователей»).
   - Каждое разрешение имеет код (`code`) и описание (`description`).
   - Примеры кодов: `view_posts`, `delete_users`, `edit_profile`.

4. **Resource (Ресурс)**  
   - Объект системы, к которому регулируется доступ (например, «посты», «комментарии», «роли»).
   - Каждый ресурс имеет имя (`name`) и описание (`description`).
   - Примеры: `posts`, `comments`, `roles`.

5. **AccessRule (Правило доступа)**  
   - Связывает роль, ресурс и разрешение, определяя, разрешено ли действие.
   - Содержит поле `is_allowed` (`True` - разрешено, `False` - запрещено).
   - Уникальность: комбинация `(role, resource, permission)`.

6. **UserRole (Связь пользователя и роли)**  
   - Ассоциирует пользователя с одной или несколькими ролями.
   - Позволяет пользователю наследовать все разрешения, назначенные его ролям.

---

## Принцип работы системы доступа

1. **Назначение ролей пользователям**  
   - При регистрации пользователю автоматически назначается роль `user`.
   - Администраторы могут назначать/менять роли через API или админ‑панель.

2. **Определение разрешений для ролей**  
   - Для каждой роли создаются правила доступа (`AccessRule`), которые указывают:  
     - К какому ресурсу (`Resource`) применяется правило.  
     - Какое действие (`Permission`) разрешается или запрещается.  
     - Для какой роли (`Role`) действует правило.  
   - Пример: роль `admin` имеет разрешение `view_posts` для ресурса `posts`.

3. **Проверка доступа при запросе**  
   - При обращении к API система:  
     1. Определяет текущего пользователя (через JWT‑токен в заголовке `Authorization`).  
     2. Получает все роли пользователя.  
     3. Проверяет, существует ли правило `AccessRule`, разрешающее запрошенное действие (`permission`) для указанного ресурса (`resource`) в рамках одной из ролей.  
     4. Если правило найдено и `is_allowed = True` - доступ разрешен. Иначе - отклонен.

## Примеры правил доступа

### Роль `admin` (администратор)
- **Ресурсы**: `posts`, `users`, `roles`, `access_rules`, `comments`, `permissions`.  
- **Разрешения**:  
  - `view_posts`, `create_posts`, `edit_posts`, `delete_posts` (для `posts`).  
  - `view_users`, `edit_users`, `delete_users` (для `users`).  
  - И другие.

### Роль `user` (обычный пользователь)
- **Ресурсы**: `posts`, `users`, `comments`.  
- **Разрешения**:  
  - `create_posts`, `view_posts` (для `posts`).  
  - `edit_profile` (для `users` - только свой профиль).  
  - `create_comments`, `view_comments` (для `comments`).

## Механизмы контроля доступа в API

1. **Middleware `CustomAuthMiddleware`**  
   - Проверяет JWT‑токен в заголовке `Authorization`.  
   - Аутентифицирует пользователя и привязывает его к запросу (`request.user`).  
   - Исключает проверку для публичных эндпоинтов (регистрация, вход, выход).


2. **Permission-классы DRF**  
   - `IsAuthenticated` - проверяет, что пользователь авторизован.  
   - `HasPermission` - проверяет наличие соответствующего `AccessRule` для запрошенного действия и ресурса.  
     - Использует `resource_code` и `permission_code` из URL или контекста запроса.


3. **View-классы с проверкой прав**  
   - В каждом API‑эндпоинте явно указываются:  
     - `resource_code` (ресурс, к которому обращается запрос).  
     - `permission_code` (действие, которое пытается выполнить пользователь).  
   - Пример:  
     ```python
     class PostDetailView(APIView):
         resource_code = 'posts'
         permission_code = 'edit_posts'  # для метода PUT
     ```

## Инициализация данных

При запуске системы (через `init_db.py`) создаются:
- Роли: `admin`, `user`.  
- Разрешения: все перечисленные в `permissions_data`.  
- Ресурсы: все перечисленные в `resources_data`.  
- Правила доступа: для `admin` и `user` согласно предопределенным спискам (`admin_rules`, `user_rules`).  
- Администратор по умолчанию: `admin@example.com` с паролем `adminpass123`.

---

## Администрирование

- **Через админ‑панель Django**  
  - Можно управлять ролями, разрешениями, ресурсами и правилами доступа.  
  - Доступно редактирование пользователей и их ролей.  
- **Через API**  
  - Эндпоинты для CRUD ролей, разрешений, правил доступа.

---

## Ключевые особенности

- **Гибкость**: новые роли, ресурсы и разрешения можно добавлять без изменения кода.  
- **Масштабируемость**: правила доступа легко модифицировать через админ‑панель или API.  
- **Безопасность**: пароли хранятся в зашифрованном виде (bcrypt), аутентификация через JWT.  
- **Прозрачность**: все правила доступа явно прописаны и видны в админ‑панели.

## Используемые технологии

- **Python 3.10.11**
- **Django 5.2.10** (веб‑фреймворк для бэкенда, ORM, аутентификации и админ‑панели)
- **DRF 3.16.1** (создание REST API с сериализацией и маршрутизацией)
- **PyJWT 2.11.0** (работа с JSON Web Tokens для аутентификации)
- **djangorestframework‑simplejwt 5.5.1** (интеграция JWT с DRF)
- **bcrypt 5.0.0** (хеширование паролей)
- **psycopg2‑binary 2.9.11** (соединение с PostgreSQL)
- **python‑decouple 3.8** (управление конфигурацией через .env‑файл)
- **PostgreSQL 16** (СУБД)

## Локальный запуск проекта
- Клонируйте репозиторий и перейдите в папку проекта:

```bash
git git@github.com:ivanlbdv/myauth.git
cd myauth
```
- Создайте и активируйте виртуальное окружение:

```bash
python -m venv venv
source venv/Scripts/activate
```
- Обновите пакетный менеджер:

```bash
python -m pip install --upgrade pip
```
- Установите модули из файла requirementst.txt:

```bash
pip install -r requirements.txt
```

- Создайте файл .env и добавьте в него:

```   
SECRET_KEY=your_secret_key
DEBUG=True
DB_NAME=auth_db
DB_USER=your_user
DB_PASSWORD=your_password
DB_HOST=localhost
DB_PORT=5432
```

- Выполните миграции:

```bash
python manage.py migrate
```

- Создайте суперпользователя:

```bash
python manage.py createsuperuser
```

Следуйте инструкциям в терминале для ввода email и пароля.

Заполните базу тестовыми данными:

```bash
python init_db.py
```

- Запустите сервер разработки:

```bash
python manage.py runserver
```

Проект будет доступен по адресу: http://127.0.0.1:8000/

## Тестирование системы

В проекте реализованы следующие тесты для проверки корректности работы системы аутентификации и авторизации:

### Тесты регистрации пользователей

* **`test_user_registration_assigns_user_role`** — проверка назначения роли `user` при регистрации нового пользователя. Убеждается, что после успешной регистрации пользователь автоматически получает базовую роль.
* **`test_registration_with_existing_email_fails`** — проверка отказа в регистрации при использовании существующего email. Гарантирует уникальность email‑адресов в системе.
* **`test_input_validation_on_registration`** — проверка валидации входных данных при регистрации (некорректный email, короткий пароль, пустые обязательные поля и т. д.). Проверяет обработку ошибок на уровне API.

### Тесты аутентификации (логин/выход)

* **`test_user_login_success`** — проверка успешного логина и получения JWT‑токена. Убеждается, что корректный пользователь получает валидный токен доступа.
* **`test_login_with_invalid_credentials_fails`** — проверка отказа при попытке логина с неверными учетными данными (несуществующий email или неверный пароль).
* **`test_login_with_inactive_user_fails`** — проверка отказа в доступе для неактивного пользователя. Гарантирует, что заблокированные или не активированные аккаунты не могут войти в систему.
* **`test_logout_success`** — проверка корректного выполнения логаута и аннулирования доступа. Убеждается, что после выхода пользователь теряет доступ к защищенным эндпоинтам.

### Тесты JWT‑токенов

* **`test_jwt_token_structure`** — проверка структуры и содержания JWT‑токена: наличие обязательных claims (`user_id`, `exp`, `iat`, `jti`, `token_type`), корректность срока действия, соответствие данных пользователя.
* **`test_middleware_valid_jwt`** — проверка успешной аутентификации с валидным JWT. Убеждается, что middleware корректно обрабатывает токен и устанавливает `request.user`.
* **`test_middleware_expired_jwt`** — проверка отказа с просроченным JWT. Гарантирует, что токены с истекшим сроком действия не дают доступа.
* **`test_middleware_invalid_jwt`** — проверка отказа с неверным JWT (неверная подпись или поврежденный токен).
* **`test_middleware_missing_authorization_header`** — проверка поведения при отсутствии заголовка `Authorization`. Убеждается, что запросы без токена отклоняются.
* **`test_middleware_invalid_authorization_format`** — проверка обработки неверного формата заголовка `Authorization` (например, отсутствует префикс `Bearer`).

### Тесты авторизации и прав доступа

* **`test_has_permission_allowed`** — доступ разрешен: у пользователя есть соответствующее правило доступа (`AccessRule` с `is_allowed=True`).
* **`test_has_permission_denied_by_rule`** — доступ запрещен: правило доступа явно запрещает доступ (`is_allowed=False`).
* **`test_has_permission_no_rules`** — доступ запрещен: нет правил доступа для роли пользователя. Проверяет поведение по умолчанию (запрет при отсутствии явного разрешения).
* **`test_has_permission_user_not_authenticated`** — доступ запрещен: пользователь не авторизован (`AnonymousUser`). Гарантирует защиту от неавторизованных запросов.
* **`test_has_permission_resource_not_found`** — доступ запрещен: ресурс с указанным кодом не найден в системе.
* **`test_has_permission_permission_not_found`** — доступ запрещен: разрешение с указанным кодом не существует.
* **`test_middleware_has_permission_no_roles`** — доступ запрещен: у пользователя нет ролей/групп/специальных прав. Проверяет базовый сценарий для «пустого» пользователя.

### Дополнительные тесты middleware

* **`test_middleware_exempt_paths`** — проверка пропуска публичных эндпоинтов (регистрация, логин, логаут) без аутентификации. Убеждается, что middleware не блокирует открытые маршруты.
* **`test_middleware_user_not_found`** — JWT валиден, но пользователь не найден в БД (например, удален после выдачи токена). Гарантирует отказ в доступе для несуществующих пользователей.
* **`test_middleware_exception_handling`** — обработка неожиданных исключений в middleware. Проверяет устойчивость системы к ошибкам внутри middleware (например, сбои в проверке прав).

---

**Примечание:** Все тесты используют изоляцию данных через `setUpTestData` и `setUp`, что гарантирует чистоту окружения для каждого запуска. Для имитации внешних зависимостей (например, проверки прав) применяются моки (`patch.object`).
